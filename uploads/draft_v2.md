### Кейс 1: JD-Gateway — единая точка правды для 160+ ВМ

**Задача:** На проекте Jet Detective 160+ виртуальных машин на тестовых стендах. Информация о них размазана по Confluence-таблицам, головам инженеров и vSphere-консолям. Никто не знает точную картину: какие ВМ живы, кому принадлежат, когда истекает TTL.

**Что сделал:** Разработал JD-Gateway на Python — веб-приложение с REST API, которое стало единственным источником правды. Интеграция с vSphere через API для полного lifecycle ВМ (создание, клонирование, удаление, управление питанием, сетями, DNS). Dynamic inventory для Ansible и service discovery для Prometheus. TTL-система с автоматическими email-напоминаниями. Web UI для команды.

**Что изменилось:** Вместо ручного создания ВМ через vSphere UI — API-вызов или клик в интерфейсе. Вместо «спроси Макса» — self-service. Ansible и Prometheus автоматически подхватывают новые хосты. TTL не дают стендам превращаться в зомби.

**Чему научился:** Что «единая точка правды» — это не архитектурный паттерн, а культурное решение. Система работает, только если все согласились через неё ходить. Техническая часть — 30%, остальное — убеждение и удобство.

### Кейс 2: Мониторинг с нуля для антифрод-платформы

**Задача:** Jet Detective — продукт без мониторинга. Проблемы обнаруживались, когда кто-то жаловался или падал стенд. Нет метрик, нет алертов, нет истории.

**Что сделал:** Построил полный стек: Prometheus для сбора метрик, Grafana для визуализации (дашборды для инфраструктуры, PostgreSQL, приложений), Thanos для долгосрочного хранения. Настроил алертинг с гибкой системой уведомлений. Добавил OpenSearch + Logstash для централизованных логов.

**Что изменилось:** Проблемы видны до того, как кто-то пожалуется. Дашборды стали основным инструментом диагностики для всей команды. Thanos хранит историю метрик, что позволяет анализировать тренды и планировать ресурсы.

**Чему научился:** Мониторинг — это не «поставить Prometheus». Это дашборд, который конкретный человек открывает каждое утро. Если дашборд не отвечает на вопрос «всё ли ок?» за 5 секунд — он бесполезен.

### Кейс 3: Jenkins как основа CI/CD на enterprise-проекте

**Задача:** Сложный продукт, много компонентов, несколько ОС, куча стендов. Нужен CI/CD, который позволяет собирать, деплоить и тестировать всё это надёжно и быстро.

**Что сделал:** Спроектировал комплексные Jenkins pipeline'ы для полного цикла. Параллельные сборки для ускорения. Динамическое создание воркеров в vSphere (не держим простаивающие машины — поднимаем по запросу). Active Choice UI, чтобы команда могла собирать и деплоить без чтения документации. Встроил автотесты в PR pipeline. Внедрил OWASP Dependency Check для безопасности. Миграция Bitbucket → GitLab с сохранением всех интеграций.

**Что изменилось:** Команда деплоит сама — без очереди к DevOps. PR не мержится без прохождения тестов. Зависимости проверяются на уязвимости автоматически. Сборки стали быстрее за счёт параллелизма.

**Чему научился:** Jenkins — мощный, но опасный инструмент. Без жёсткой дисциплины pipeline'ы превращаются в нечитаемые Groovy-скрипты. Если что-то падает — это повод разобраться, а не обойти.

### Кейс 5: Полтора месяца в LLM-зоопарке [site]

**Задача:** Понять, можно ли с помощью LLM-агентов реально снять когнитивную нагрузку — не в теории, а в ежедневной инженерной работе. Не «попробовать ChatGPT», а построить рабочую экосистему.

**Что сделал:** 299 часов за полтора месяца. Собрал экосистему из 16 агентов на двух Mac mini. Попробовал: OpenClaw (оркестрация агентов), Claude Code, Codex (GPT 5.4), Gemini, Hermes, локальные модели (Gemma, Qwen), системы памяти (Honcho, ollama-bge-m3), управляющие слои (Multica, Paperclip).

**Что выкинул:**
- 16 параллельных агентов — зашкаливающая когнитивная нагрузка, обратный эффект
- Локальные модели на 16GB Mac mini — слишком слабые для реальной пользы
- Multica, Paperclip, Honcho — оверинжиниринг, не интегрировались в рабочий процесс

**Что оставил (3 инструмента):**
- Claude Code — для кода и файлов. «После его правок НЕ хочется всё переписать»
- Клоя (OpenClaw) — персональный агент с памятью в Telegram
- Hermes — для сложных задач, требующих глубокого анализа

**Чему научился:** Главная ловушка LLM — «когда у тебя молоток, всё вокруг гвозди». 16 агентов давали эндорфины, но не результат. Путь от усложнения к упрощению — тот же принцип, что и в DevOps: сначала порядок, потом автоматизация. Инструмент должен снимать нагрузку, а не создавать новый зоопарк.

## 6. Принципы [site]

> То, как я подхожу к работе.

- **Сначала порядок, потом автоматизация.** Автоматизировать хаос — значит получить автоматизированный хаос.
- **Backup перед рискованными изменениями.** Всегда. Без исключений. Даже если «точно ничего не сломается».
- **Инструмент должен упрощать жизнь, а не создавать новую религию.** Если тул требует полставки на поддержку — это не инструмент, это проект.
- **Живое важнее стерильного.** Работающая система с небольшим техдолгом лучше идеальной архитектуры, которую никто не смог довести до прода.
- **Документация — часть работы.** Если это не записано — этого не существует. Через полгода ты сам не вспомнишь, почему так сделал.
- **set -e, никаких скипов.** Если что-то падает — разберись, а не обойди.
